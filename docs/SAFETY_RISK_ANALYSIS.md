# Safety / Risk Analysis

## 1. Метод

Качественная оценка по вероятности (Low/Med/High) и влиянию (Low/Med/High).

## 2. Реестр рисков

| ID | Риск | Вероятность | Влияние | Митигирующие меры | Связь с требованиями |
|---|---|---|---|---|---|
| R-01 | Потеря RX байта при высоком потоке (mailbox 1-byte) | Med | Med | Документировать ограничение; при необходимости заменить на ring buffer RX | SRS-F-06 |
| R-02 | Неверный debounce вызывает ложные события | Low | Med | Unit tests + фиксированные тайминги | SRS-F-02 |
| R-03 | Переполнение TX ring buffer | Med | Low/Med | Событие может быть пропущено; мониторинг и настройка размера буфера | SRS-F-04 |
| R-04 | Блокирующий UART TX увеличивает латентность цикла | Med | Med | Ограничение частоты отправки через scheduler | SRS-F-05 |
| R-05 | Ошибка в критических секциях (ISR race) | Low | High | Абстракция `critical_section_ops_t`, review, unit tests для буфера | SRSw-CON-01 |
| R-06 | Непредсказуемость при изменении `F_CPU` | Med | Med | Явная конфигурация и проверка таймингов в release | SRS-NF-04 |

## 3. Остаточные риски

- `R-01` остаётся частично открытым до внедрения RX ring buffer.
- `R-04` допустим для текущего трафика и требований.

## 4. Рекомендации

1. При росте RX нагрузки перейти с mailbox на RX ring buffer.
2. Добавить watchdog стратегию и политику recovery.
3. Ввести HIL smoke-тест в release pipeline.
